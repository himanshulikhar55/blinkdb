BlinkDB
Note
BlinkDB is a project developed for educational purposes and is not intended for production use.

Overview

BlinkDB is an efficient in-memory key-value store that leverages both in-memory and on-disk storage to provide quick data access and high write speeds. The project employs an in-memory hashtable alongside disk storage for evicted entries (when the main memory starts to fill up), ensuring scalability when the dataset grows large. Additionally, it integrates a bloom filter and a sparse index to optimize read operations and disk lookups.

Architecture

The architecture of BlinkDB consists of:

  • In-Memory Hashtable: Utilizes open addressing with double hashing to store key-value pairs until the load factor reaches 70%.
  • Resizing and Rehashing: When feasible (i.e., if memory usage is below 1GB), the hashtable is resized and rehashed to accommodate more entries.
  • Disk Storage Eviction: If resizing is not possible, 50% of the in-memory entries are evicted to disk storage to free up memory.
  • Disk Storage: Maintains evicted entries and is optimized using a sparse index for faster file lookups.
  • Bloom Filter: Quickly rules out non-existent keys to reduce unnecessary disk accesses during read operations.

Operations

BlinkDB provides the following core functionalities:

  • SET: Insert or update a key-value pair in the store.
  • GET: Retrieve the value associated with a given key, first by checking in-memory storage, then, if needed, falling back to disk storage.
  • DEL: Delete a key-value pair from the in-memory store.
  • print: Output the current contents of the in-memory store to the console.

Cleanup

Upon termination, BlinkDB ensures that both in-memory and disk data are cleared, maintaining data integrity and resource management. This was achieved after much deliberation and hours of testing using Valgrind and GDB.
